Protocol Optimiser — System Architecture

Six Python modules, three external services, one rules.json file connecting them all.

External Services

Data source
NCBI PubMed + PMC (E-utilities API)
The US National Library of Medicine's free API for searching and retrieving biomedical literature, including PubMed Central full-text when Open Access.
esearch — find PMIDs matching a query
↓
esummary — lightweight metadata (journal, date, PMCID)
↓
efetch db=pubmed — full abstract text (XML)
↓
efetch db=pmc — Methods section from full-text (XML, OA only)
Four separate API endpoints are called in sequence. PubMed requests are chunked in groups of 20, PMC in groups of 10, to respect NCBI rate limits. PMC Methods enrichment only succeeds for Open Access papers.
AI service
OpenRouter API
Unified gateway to large language models (Claude, Gemini). Used for two distinct purposes in this system.
Rule generation — synthesise PubMed evidence + optional user reference into structured TE/TR benchmarks
↓
Semantic mapping — translate vendor series names to canonical clinical labels
Rule generation uses Claude (complex reasoning); semantic mapping uses Gemini Flash (fast, cheap single-question lookup). API key stored in environment variable, never committed to code.

Module 1 — RAG Pipeline

Python module
generate_rules_with_pubmed.py
The evidence-to-benchmark pipeline. Searches PubMed, enriches with PMC full-text where available, optionally merges an OEM vendor reference document and/or a user-provided site reference document, scores abstracts, and asks an LLM to produce structured rules.
_multi_strategy_search()
Runs three complementary PubMed queries — clinical guidelines (via MeSH medical headings), technical parameters (papers mentioning TE/TR/flip angle), and protocol optimization (scan time, acceleration). Each returns up to 20 PMIDs. Results are merged and deduplicated into a single pool of up to ~60 papers. Date filter uses a dynamic range from 2018 to the current calendar year.
↓
get_pubmed_ids() → fetch_pubmed_summaries()
Calls PubMed esearch for IDs, then esummary for lightweight metadata: journal name, publication date, PMCID (for full-text access), and whether the journal is on the gold-standard list (Radiology, AJNR, JMRI, European Radiology, RadioGraphics, Investigative Radiology, AJR). Gold-journal matching uses start-anchored word-boundary regex to prevent false positives (e.g. "Pediatric Radiology" does not match "Radiology").
↓
fetch_pubmed_abstracts_by_pmid()
Fetches full abstract text via PubMed efetch (XML format). Requests are split into chunks of 20 PMIDs to stay within NCBI rate limits for larger pools.
↓
_fetch_pmc_methods()
For papers that have a PMCID (indicating PMC Open Access availability), fetches the full-text XML and extracts text from Methods/Materials sections. This text is appended to the abstract, enriching both the density scorer and LLM context with the parameter tables that rarely appear in abstracts alone. Chunked in groups of 10.
↓
_parameter_density()
Counts regex matches for 14 MRI-specific patterns in each abstract (+ Methods when available): TE, TR, TI (with flexible delimiters), context-guarded ms and mm values (requiring MRI terms like TE/TR/slice/FOV nearby), flip angle, FOV, matrix size, slice thickness, b-value, voxel size, NEX, NSA, and bandwidth. A paper scoring 5 means it mentions 5 distinct technical terms — it likely contains a parameter table. A paper scoring 0 is purely clinical.
↓
select_top_five_pmids()
Composite ranking with four tiers: (1) gold-standard journal first, (2) published 2018 or later, (3) highest parameter density, (4) most recent date. The top 5 papers that survive this ranking and have non-empty abstracts are selected.
↓
OEM document merge (via oem_registry.py)
When the user selects a vendor (Philips, Siemens, GE, Canon) and an OEM reference document, the curated PDF is read, text-extracted, and prepended with weighting instructions: rank below gold-standard PubMed literature but above model training knowledge. OEM documents fill parameter gaps that abstracts often omit for vendor-specific implementations.
↓
_extract_text_from_pdf_bytes() + _prepare_supplementary_text()
When the user attaches a site reference document (PDF or TXT) or pastes an excerpt, text is extracted (pypdf for PDFs) and, if longer than 24k chars, chunked into 5k-char windows that are ranked by parameter density + protocol keyword relevance. The highest-signal slices are selected up to the size cap. OEM and site documents are merged into a combined supplement with source-type labels and weighting instructions.
↓
_build_user_rag_content()
Assembles the LLM prompt: each paper is labeled with its PMID, journal, gold-standard status, parameter density score, and source depth (abstract vs abstract+methods). If the user selected 1.5T, a hardware advisory note is prepended. The combined OEM + site supplement (if any) is appended with source-type labels and weighting instructions.
↓
OpenRouter LLM call → validate_rules_schema()
The LLM receives the system prompt (including the GRADE-aligned evidence rubric, which accounts for user-provided references) plus the assembled evidence. It returns a JSON object containing clinical_rationale (summary, evidence_strength, key_changes), study_rules, and series_protocols with TE/TR ranges and target durations. The JSON is validated against the expected schema before being written to disk.
↓
→ rules.json
The single source of truth for all subsequent audits. Contains: acceptable TE/TR ranges per sequence, target acquisition durations, clinical rationale, and GRADE evidence strength. Overwritten each time a sync runs.

Module 2 — Non-PubMed Rule Generation

Python module
generate_rules.py
Direct LLM rule generation without PubMed. Used when the user wants to generate rules from a free-text protocol description rather than live literature.
generate_protocol_rules(prompt, scanner_tesla)
Sends a protocol description (e.g. "Standard MS Brain MRI protocols") directly to the LLM with the same GRADE-aligned schema and evidence rubric. If scanner_tesla is 1.5, the 1.5T advisory is prepended. Useful for quick benchmarking without waiting for PubMed search.
↓
→ rules.json
Same output format and validation as the PubMed path. The evidence_strength will typically be "Low" or "Moderate" since the LLM is relying on training knowledge rather than cited abstracts.
Also provides (shared utilities)
OPENROUTER_API_KEY validate_rules_schema() _extract_json_object() write_rules_file()
These constants and functions are imported by other modules (rule_engine.py, generate_rules_with_pubmed.py) to avoid duplication. This module also runs load_dotenv() on import, ensuring environment variables from the .env file are available system-wide.

Module 3 — DICOM Parser

Python module
dicom_parser.py
Reads a single DICOM file and extracts only the physics metadata needed for auditing. Deliberately ignores all patient-identifying information.
parse_dicom(path)
Opens the .dcm file using pydicom with stop_before_pixels=True (never loads image data). Uses safe extraction helpers (_safe_str, _safe_float_ms, _safe_positive_int, _safe_positive_float, _pixel_spacing_mm) that return None instead of crashing on missing or malformed tags.
↓
Extracted tags (physics only — no PHI)
Study Description (0008,1030) Series Description (0008,103E) TE (0018,0081) TR (0018,0080) Flip Angle (0018,1314) Inversion Time (0018,0082) Acquisition Duration (0018,9073) Slice Thickness (0018,0050) Spacing Between Slices (0018,0088) Pixel Spacing (0028,0030) Matrix Rows (0028,0010) Matrix Columns (0028,0011) Number of Averages (0018,0083) Magnetic Field Strength (0018,0087) Manufacturer (0008,0070) Manufacturer Model (0008,1090)
These are technical sequence parameters that describe how the MRI scanner was configured — not who was scanned. Patient Name, DOB, and all other identifiers are never read. FOV is derived from Matrix × Pixel Spacing when both tags are present.
↓
_duration_heuristic_ms()
When tag 0018,9073 (AcquisitionDuration) is absent — which is common in older DICOM files — a rough estimate is computed from TR × Number of Averages × echo-train scaling. Labeled as "heuristic estimate" in the UI so the user knows it's approximate.
↓
→ Returns dict with all physics fields (or None for missing tags)

Module 4 — Rule Engine

Python module
rule_engine.py
The core audit logic. Compares a parsed DICOM against rules.json and produces the complete audit result: drift status, efficiency score, clinical grade, and hardware warnings.
load_rules() — reads rules.json from disk
↓
evaluate(parsed_data, rules)
The main entry point. Receives the DICOM physics dict from the parser and the rules dict. Orchestrates all checks below and returns a single result dict.
↓
Step 1 — Series matching with confidence rating
First attempts an exact string match between the DICOM's SeriesDescription and the keys in rules.json (confidence: high). If no match is found, calls map_series_semantic() which sends the series name and all candidate keys to OpenRouter (Gemini Flash) and asks: "Which canonical label is the closest clinical match?" (confidence: medium). Each match also generates a plain-language benchmark rationale explaining why this comparison was chosen. This handles vendor naming inconsistencies like "T1 3D 0" → "3D T1 MPRAGE SAG".
↓
Step 2 — Physics drift check: _check_timing()
Compares actual TE and TR values from the DICOM against the min/max ranges in rules.json for the matched series. If either value falls outside the range, it's flagged as "PROTOCOL DRIFT DETECTED" with the expected range and actual value shown. If both are within range, the result is PASS.
↓
Step 3 — Efficiency scoring: _compute_efficiency_and_revenue()
Compares the scan's actual acquisition duration against the target_duration_ms from rules.json. Score = 100 if at or faster than target; decays as the scan gets slower. If the scan is more than 20% slower than the benchmark, a "Revenue Opportunity" alert is generated with an illustrative throughput gain estimate.
↓
Step 4 — Hardware signal warning: _hardware_signal_warning()
Only fires when physics PASS and scanner is 1.5T. Checks whether TE and/or TR sit in the bottom 25% of the acceptable range — indicating parameters that were likely designed for 3T's higher signal-to-noise and may produce grainy images on 1.5T hardware.
↓
Step 5 — Clinical grade: _clinical_grade()
Combines physics status and efficiency score into a single letter grade. A = physics pass + efficient (score ≥ 80). B = physics pass + no data or moderate efficiency (score ≥ 65). C = physics pass + slow (score ≥ 45). F = physics fail (any TE/TR drift detected).
↓
→ Returns complete audit result dict
A single dictionary containing: status (pass/fail/error), messages, mapped_series_key, mapping_method, match_confidence, benchmark_rationale, canonical_sequence_label, efficiency_score, efficiency_target_ms, efficiency_actual_ms, efficiency_ratio, revenue_opportunity, revenue_message, hardware_signal_warning, and clinical_grade. This is consumed directly by the Streamlit UI.

Module 5 — Streamlit Dashboard

Python module
app.py
The web interface. A single-page Streamlit application that connects all modules into an interactive audit workflow.
Sidebar — Guideline Management
Protocol text input, scanner selector (1.5T / 3.0T / Not specified), OEM vendor document selector (manufacturer auto-detected from DICOM, with curated documents for Philips, Siemens, GE, Canon), optional site reference document upload (PDF or TXT) with a text-area alternative for pasting excerpts, and the "Sync with PubMed" button. When clicked, extracts text from OEM and site documents, merges them with source-type labels and weighting, then calls generate_rules_from_pubmed() which runs the full RAG pipeline. After sync, displays the list of selected PubMed sources with gold-star indicators, publication years, direct PMID links, plus OEM and user document rows if included.
↓
Clinical Rationale & Evidence panel
Reads clinical_rationale from rules.json and displays: an executive summary (2-4 sentences), a GRADE-aligned evidence strength badge (color-coded High/Moderate/Low with an explanation of what the rating means), a recent literature changes section, and a PubMed sources table showing Year, Journal, Impact Factor star, Parameter Density score, and PMID link for each paper used. When evidence is Low, an actionable tip is shown guiding the user to refine their query, attach a reference PDF, or verify against local protocols.
↓
File uploader — drag-and-drop .dcm
Accepts a single DICOM file. Writes it to a temporary path, calls parse_dicom() to extract physics tags, then calls evaluate() to run the full audit. The temp file is deleted immediately in a finally block — even if an error occurs. A privacy notice is displayed below the uploader.
↓
Sequence parameters display
Five metric columns: Scan hardware (e.g. "3.0T"), Series description, TE (ms), TR (ms), and Acquisition duration (explicitly labeled as "from DICOM" or "heuristic estimate").
↓
Sequence identity row — Flip Angle, TI, Benchmark Protocol
A second compact row showing flip angle (degrees), inversion time TI (ms), and the matched benchmark protocol label. These help the radiologist verify whether the sequence type was correctly identified — flip angle and TI are essential for distinguishing IR-prepared volumes from spoiled gradient-echo sequences.
↓
Acquisition context (expandable) — spatial resolution details
Expandable section showing: matrix rows/columns, number of averages, slice thickness (mm), slice spacing (mm), voxel size (row × col × slice, in mm), FOV (derived from matrix × pixel spacing, in mm), and pixel spacing. Missing tags display as "—". Spatial context lets the radiologist judge whether a protocol compromise happened in resolution rather than just timing.
↓
Series mapping with confidence indicator
Displays how the scanner's series label was mapped to a benchmark: exact match (high confidence, green indicator) or AI semantic mapping (medium confidence, yellow indicator). A plain-language benchmark rationale caption explains why the comparison was chosen. This transparency helps radiologists judge whether a drift finding is clinically meaningful or a mapping artefact.
↓
Clinical audit strip
Four metric columns: Clinical Grade (A/B/C/F with color emoji), Efficiency Score (0-100), Target time (from PubMed benchmark), and Observed duration (with % delta). Revenue opportunity and hardware signal warnings appear as alert banners when triggered.
↓
Protocol drift verdict
A green PASS box or red FAIL box showing the final physics result. On failure, the specific TE/TR expected range and actual value are displayed so the radiologist knows exactly which parameter drifted and by how much.

Module 6 — OEM Document Registry

Python module
oem_registry.py
Curated registry of MRI-relevant OEM reference documents for RAG enrichment. Maps vendor names to available protocol guides stored in the oem_docs/ directory.
OEM_DOCUMENT_REGISTRY — structured list of vendor documents
Each entry contains: unique ID, display label, PDF filename, manufacturer (Philips, Siemens, GE, Canon), category (protocol_guide, technology, system_overview), body region, supported field strengths, whether it contains explicit parameters, and a description. Currently contains documents covering SmartExam workflows, AIR Recon DL, AiCE deep learning reconstruction, Compressed Sensing, and vendor-specific protocol tables.
↓
normalize_manufacturer() — DICOM Manufacturer tag alias mapping
Maps common DICOM Manufacturer tag values (e.g. "PHILIPS MEDICAL SYSTEMS", "GE MEDICAL SYSTEMS", "SIEMENS HEALTHINEERS", "TOSHIBA") to canonical manufacturer names used by the registry. This allows auto-detection of the scanner vendor from the uploaded DICOM file.
↓
get_docs_for_manufacturer() / resolve_doc_path()
Filters available documents by manufacturer and resolves the absolute path to each PDF in the oem_docs/ directory. Used by the Streamlit sidebar to populate the OEM document selector and by the RAG pipeline to read the selected document.

Data Flow Summary

① User clicks "Sync" → app.py extracts text from optional OEM vendor document (via oem_registry.py) and/or site reference document → merges with source-type labels and weighting → calls generate_rules_with_pubmed.py → which calls PubMed API (3 searches) → fetches abstracts → fetches PMC Methods (where OA) → merges OEM + site supplement → scores density → ranks → calls OpenRouter LLM → writes rules.json
② User uploads .dcm → app.py calls dicom_parser.py → extracts physics tags (TE, TR, FA, TI, slice thickness, pixel spacing, voxel size, FOV, duration, field strength, manufacturer) → deletes temp file → auto-detects manufacturer for OEM selector → calls rule_engine.py with parsed data + rules.json
③ rule_engine.py → matches series (exact or via OpenRouter semantic mapping) with confidence rating → checks TE/TR drift → scores efficiency → checks hardware → assigns grade → generates benchmark rationale → returns result dict
④ app.py renders the result: rationale panel (with Low-evidence tip when applicable), sequence parameters + identity row (FA, TI, benchmark protocol), spatial resolution context (voxel, FOV, slice thickness), series mapping with confidence indicator, audit strip, and drift verdict
